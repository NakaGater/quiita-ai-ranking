# Feature Specification: Qiita AI Ranking

**Feature Branch**: `001-ai-ranking`
**Created**: 2026-03-09
**Status**: Draft
**Input**: User description: "Qiita APIからAI/機械学習関連の記事データを取得し、トレンドランキングを計算してWebページとして表示するランキングサイト"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ランキング一覧の閲覧 (Priority: P1)

ユーザーがサイトにアクセスすると、AI/機械学習関連のQiita記事がスコア順にランキング表示される。各記事にはタイトル、著者名、スコア、タグ一覧、投稿日が表示される。ユーザーは記事タイトルをクリックすることで元のQiita記事ページに遷移できる。

**Why this priority**: ランキングの閲覧はサイトの核心機能であり、これなしではサイトの存在意義がない。

**Independent Test**: サイトにアクセスし、ランキングが表示され、記事リンクが機能することを確認する。

**Acceptance Scenarios**:

1. **Given** サイトにアクセスした, **When** ページが読み込まれた, **Then** AI関連記事がスコア降順でランキング表示される
2. **Given** ランキングが表示されている, **When** 記事タイトルをクリックした, **Then** 元のQiita記事ページに遷移する
3. **Given** ランキングが表示されている, **When** 各記事を確認した, **Then** タイトル、著者名、スコア、タグ、投稿日が表示されている

---

### User Story 2 - 記事データの自動収集 (Priority: P1)

システムがQiita API v2を使用して、AI/機械学習関連タグ（AI, 機械学習, MachineLearning, LLM, ChatGPT, GPT, 深層学習, DeepLearning, NLP, 自然言語処理, 生成AI, Transformerなど）の記事を検索・取得する。取得した記事データはJSONファイルとしてローカルに保存される。

**Why this priority**: データ収集はランキング表示の前提条件であり、これがなければランキングを計算できない。

**Independent Test**: データ取得スクリプトを実行し、JSON形式で記事データが保存されることを確認する。

**Acceptance Scenarios**:

1. **Given** Qiita APIトークンが設定されている, **When** データ取得バッチを実行した, **Then** AI関連タグの記事データがJSON形式で保存される
2. **Given** APIからデータを取得した, **When** 保存されたデータを確認した, **Then** 各記事にタイトル、著者、いいね数、ストック数、タグ、URL、投稿日が含まれている
3. **Given** API呼び出し時, **When** レートリミットに達した, **Then** 適切な待機後にリトライする

---

### User Story 3 - スコアリングとランキング算出 (Priority: P1)

取得した記事データに対してスコアリングアルゴリズムを適用し、ランキング順位を算出する。スコアはいいね数、ストック数、記事の新しさ（新しい記事ほどボーナス）を加味した複合スコアで計算される。

**Why this priority**: スコアリングはデータ収集と表示を繋ぐ核心ロジックであり、ランキングの質を決定する。

**Independent Test**: サンプルの記事データに対してスコアリングを実行し、期待通りの順序でランキングされることを確認する。

**Acceptance Scenarios**:

1. **Given** 記事データが存在する, **When** スコアリングを実行した, **Then** 各記事にスコアが付与される
2. **Given** いいね数が多い記事とストック数が多い記事がある, **When** スコアを比較した, **Then** 両方の指標が反映されたスコアになっている
3. **Given** 同じいいね数・ストック数の記事が2つある, **When** 一方が新しい記事の場合, **Then** 新しい記事のスコアが高くなる

---

### User Story 4 - 日次バッチ更新 (Priority: P2)

システムが日次でデータ取得→スコアリング→サイト再ビルドの一連のパイプラインを自動実行し、ランキングを最新の状態に保つ。

**Why this priority**: 日次更新はランキングの鮮度を保つために重要だが、手動実行でもMVPとしては成立する。

**Independent Test**: バッチスクリプトを実行し、データ取得からサイトビルドまでの全工程が自動で完了することを確認する。

**Acceptance Scenarios**:

1. **Given** バッチスクリプトが設定されている, **When** スケジュール実行された, **Then** データ取得→スコアリング→ビルドが順番に実行される
2. **Given** バッチが実行された, **When** 完了後にサイトを確認した, **Then** 最新のランキングが反映されている

---

### Edge Cases

- Qiita APIが一時的に利用不可の場合、前回取得したデータでランキングを表示する
- 記事データが0件の場合、適切なメッセージを表示する
- APIレートリミットに達した場合、適切にリトライまたはスキップする
- いいね数・ストック数が全て0の記事はランキング対象外とする

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはQiita API v2を使用してAI関連タグの直近30日間の記事を検索・取得できなければならない
- **FR-002**: システムは取得した記事データをJSON形式でローカルファイルに永続化しなければならない
- **FR-003**: システムはいいね数（重み3）、ストック数（重み2）、記事の新しさ（重み1）を加味した複合スコアを算出しなければならない
- **FR-004**: システムはスコア降順で上位50件の記事ランキングを生成し、1ページに表示しなければならない
- **FR-005**: ランキングページは記事タイトル、著者名、スコア、タグ一覧、投稿日を表示しなければならない
- **FR-006**: 各記事のタイトルは元のQiita記事ページへのリンクでなければならない
- **FR-007**: システムはデータ取得→スコアリング→ビルドを一括実行するバッチスクリプトを提供しなければならない
- **FR-008**: APIトークンは環境変数で管理し、コードにハードコードしてはならない

### Key Entities

- **Article（記事）**: Qiitaから取得した記事データ。タイトル、著者名、URL、いいね数、ストック数、タグ一覧、投稿日、記事IDを持つ
- **RankedArticle（ランキング記事）**: スコアリング後の記事。Articleの全属性に加えてスコアとランキング順位を持つ
- **RankingMetadata（ランキングメタデータ）**: ランキング全体の情報。最終更新日時、記事総数、対象タグ一覧を持つ

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがサイトにアクセスしてから3秒以内にランキング一覧が表示される
- **SC-002**: ランキングには最低50件のAI関連記事が含まれる
- **SC-003**: ランキングのデータは24時間以内に更新される
- **SC-004**: すべての記事リンクが正しいQiita記事ページに遷移する
- **SC-005**: モバイルデバイスでもランキングが正しく表示される

## Clarifications

### Session 2026-03-09

- Q: ランキング対象記事の取得期間は？ → A: 直近30日間の記事のみ対象
- Q: ランキング表示件数は？ → A: 上位50件を1ページに表示（ページネーションなし）
- Q: スコアリングの重み配分は？ → A: いいね:ストック:新しさ = 3:2:1（いいね重視）

## Assumptions

- Qiita APIのアクセストークンはユーザーが事前に取得済みである
- Qiita API v2の無料枠（認証済みで1時間あたり1000リクエスト）で十分なデータが取得可能である
- 対象タグは日本語・英語の両方をカバーする（AI, 機械学習, MachineLearning, LLM, ChatGPT, GPT, 深層学習, DeepLearning, NLP, 自然言語処理, 生成AI, Transformer）
- スコアリングの重み付けは初期実装では固定値とし、後から調整可能にする
- 閲覧数（page_views_count）はQiita API経由では取得制限がある可能性があるため、いいね数とストック数を主要指標とする
